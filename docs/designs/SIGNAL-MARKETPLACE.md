## Vision

### 10x Check
Sage Radar AI becomes the "ZuluTrade for Telegram signals" — the platform where Telegram signal providers are discovered, rated by verified performance, and consumed without anyone needing to touch Telegram directly. Admin-curated channels monitored by a dedicated admin TG account, every signal tracked and ranked with win rate/drawdown/P&L, users subscribe with one click and connect their webhook. Public marketplace drives SEO acquisition. Top providers featured, creating a flywheel: more users -> better data -> more providers -> more users.

### Platonic Ideal
A user opens Sage Radar AI for the first time — no account needed. They see a grid of signal providers with verified performance stats. They sort by win rate, filter for forex, and find a provider with 72% win rate over 90 days. They click "Subscribe", sign up, paste their SageMaster webhook URL, and they're live in under 60 seconds. Zero Telegram friction. The feeling: "Why would I ever go back to finding providers on Twitter?"

## Problem Statement (from /office-hours)

Users are losing money and time on bad Telegram signal providers. They discover providers through unreliable channels (social media, word of mouth), pay for VIP channels blind, churn through expensive trial-and-error, and ask the founder directly for recommendations. The founder IS the marketplace today, manually. No platform offers verified, platform-computed performance data for Telegram signal providers covering both forex and crypto.

## Demand Evidence

- Direct user requests: subscribers have explicitly asked for provider discovery
- Observed behavior: users ask founder/community for channel recommendations
- Pain quantified: users lose real money on trial-and-error with bad providers
- Supply exists: channels already in the system have signal_logs data for verified performance
- Competitive gap: Cornix is crypto-only, no forex marketplace exists for Telegram

## Structural Moat

Sage Radar's LLM parser creates structured data from every signal (entry, SL, TPs, direction, symbol). Every signal outcome is logged. This enables auto-verified performance tracking without provider cooperation. Providers cannot delete losing trades. No competitor has this capability.

## Key Architectural Revision (vs March 18 plan)

**Admin Telegram Account as Marketplace Listener**: Instead of piggybacking on existing user connections (fragile, no uptime guarantee), the admin's own Telegram account joins all marketplace-listed channels. This guarantees 100% signal delivery for marketplace subscribers without requiring a full Master Listener infrastructure service.

This supersedes the March 18 decision to "defer Master Listener — piggyback on user connections." The admin account approach is simpler, reliable, and sufficient for the initial provider count (5-20 channels).

### Signal Delivery Model
Admin's own Telegram account joins all marketplace-listed channels. When the admin listener picks up a signal from a marketplace channel, the workflow fan-out routes ParsedSignal to all marketplace subscribers of that channel. Subscribers never see raw Telegram messages.

## Pre-Resolved Architecture Decisions

1. **Fan-out**: Workflow fan-out (Step 2 of workflow.py), NOT listener broadcast
2. **Data privacy**: Marketplace subscribers receive ParsedSignal only, never raw TG messages
3. **Signal logs identity**: `source_type` enum column ('telegram' | 'marketplace')
4. **Feature flag**: `MARKETPLACE_ENABLED` env var controls visibility + fan-out
5. **Admin listener**: Multiple dedicated TG accounts (not personal) join marketplace channels for guaranteed delivery. If one account gets restricted, others continue. Requires new phone numbers + session setup per account.

## Implementation Details (Spec Review Fixes)

### Stats Computation Strategy
Provider performance stats (win rate, P&L, drawdown) are computed from `signal_logs` via periodic background task (every 15 minutes). The task aggregates signals per `telegram_channel_id` where `source_type = 'marketplace'`. Results cached in `marketplace_providers` stat columns (win_rate, total_pnl_pips, max_drawdown_pips, signal_count, last_computed_at). Cache invalidation: on-demand via admin dashboard refresh button, or automatic every 15 min.

**P&L definition**: Measured in pips (forex) or percentage (crypto), NOT in dollar terms — avoids dependency on subscriber lot sizes. Win rate = signals that hit any TP / total closed signals. Max drawdown = peak-to-trough decline in cumulative pip P&L (standard drawdown definition, not consecutive losses).

**Trade outcome tracking** (highest-risk implementation area):
- **V1 approach**: Count management signals from the same provider that reference the same symbol and match to the original entry signal (by symbol + time window). Parser already extracts `modify_tp`, `modify_sl`, `close` actions — these indicate trade lifecycle events.
- **TP hit detection**: When provider sends "TP1 hit" or "close partial" for the same symbol within 7 days of entry, mark original signal as partial win. "Close all" or "SL hit" marks as loss.
- **Fallback**: If no explicit close signal within 7 days, mark as "expired/unknown" and exclude from win rate calculation.
- **V1 limitation acknowledged**: Outcome tracking is proxy-based and imperfect. Some providers never send close signals. V1 stats display "N signals tracked, M outcomes verified" to communicate confidence level. SageMaster does not report back trade results.
- **V1.1 enhancement**: Add market price comparison — fetch closing prices from a price feed API to verify TP/SL hits independently of provider messages.

### Source Type Determination
`source_type` is set by the **workflow** (not the listener). When the workflow processes a signal, it checks if the originating `telegram_channel_id` exists in `marketplace_providers` (active). If yes AND the signal came from the admin listener account, `source_type = 'marketplace'`. If the same channel is both a user's personal connection and a marketplace listing, the user's personal signal is `source_type = 'telegram'` and the marketplace fan-out creates separate `signal_logs` entries with `source_type = 'marketplace'` for each subscriber. No dedup — both paths produce independent log entries.

### Fan-Out Mechanics
After parsing (workflow Step 2), if the channel is a marketplace provider:
1. Query `marketplace_subscriptions` WHERE `provider.telegram_channel_id = signal.channel_id` AND `is_active = true`
2. For each active subscription, load the associated `routing_rule` (via `routing_rule_id` FK)
3. Run mapper + dispatcher for each routing rule **sequentially** (not parallel — avoid overwhelming SageMaster webhooks). At 50+ subscribers per provider, switch to batched parallel dispatch (5 concurrent). Budget: 30-second timeout per fan-out batch.
4. Each dispatch creates its own `signal_log` entry with `source_type = 'marketplace'`
5. Per-subscriber errors are isolated — one subscriber's webhook failure does not block others
6. Failed dispatches logged with `status = 'error'`, successful with `status = 'success'`

### Subscribe Lifecycle
1. User browses `/marketplace` (no auth) → sees provider cards with stats
2. Clicks "Subscribe" → redirected to login/signup if not authenticated
3. After auth → consent screen with disclaimer ("Past performance is not indicative of future results. Sage Radar routes signals — it does not execute trades. You are responsible for your own trading decisions.")
4. User must have at least one webhook URL configured (SageMaster destination)
5. On consent → system creates: `marketplace_subscription` record, `marketplace_consent_log` entry, and a new `routing_rule` linking the provider's channel to the user's webhook destination
6. Signals begin flowing immediately via workflow fan-out

### Unsubscribe Lifecycle
1. User clicks "Unsubscribe" on `/dashboard/subscriptions`
2. `marketplace_subscription.is_active` set to `false`
3. Associated `routing_rule` is **soft-deleted** (is_active = false) — preserves signal log history
4. In-flight signals already dispatched are unaffected (fire-and-forget)
5. User can re-subscribe later — creates a new subscription + routing rule (fresh consent required)

### Marketplace-First Onboarding Flow
1. New user lands on `/marketplace` (public, no auth)
2. Browses providers, clicks "Subscribe" on one
3. Redirected to signup (email + password or Google OAuth)
4. After signup → prompted to add SageMaster webhook URL (destination setup)
5. Consent screen → subscribe → routing rule created → live
6. User's dashboard shows marketplace subscriptions. Telegram connection is optional and presented as "Bring Your Own Channels" for power users.

### Admin CRUD Endpoints (Full)
- POST   /api/admin/marketplace/providers — Create provider
- GET    /api/admin/marketplace/providers — List all providers (with stats)
- GET    /api/admin/marketplace/providers/{id} — Get provider detail
- PUT    /api/admin/marketplace/providers/{id} — Update provider
- DELETE /api/admin/marketplace/providers/{id} — Deactivate provider (soft-delete: is_active=false)
- On provider deactivation: all active subscriptions are paused (is_active=false), subscribers notified via dashboard banner. Routing rules deactivated. Subscribers can be reactivated if provider is re-listed.

### Consent Framework
Disclaimer text stored as a versioned template in code (not DB). Current version: "1.0". When version bumps, existing subscribers see a re-consent prompt on next login. `marketplace_consent_log` tracks every consent event with version. Disclaimer content:
- "Past performance is not indicative of future results"
- "Sage Radar dispatches webhook signals — it does not execute trades"
- "You are solely responsible for your trading decisions and risk management"
- "Signal delivery depends on platform availability — no uptime guarantee"

### Admin TG Account Scalability
Migration trigger: when total inbound message rate from marketplace channels exceeds 60 messages/minute sustained over 1 hour (measured via Sentry breadcrumbs in listener heartbeat). At that point, migrate to dedicated Master Listener service (separate Railway worker, separate TG session, dedicated proxy). This is a threshold, not a channel count.
